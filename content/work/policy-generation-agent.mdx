## The problem

Authoring a new compliance control at Amazon's scale used to take roughly six months end to end. The bottleneck was not novel thinking. It was sequential review.

A new control passes through legal, security, and product partners, each reviewing in turn. Every meaningful edit triggers a fresh review cycle. The drafting itself rarely takes more than a week of active work. The rest is calendar time spent in queues and re-reviews. By the time a control ships, the regulation it addresses may have moved on. At the scale we needed, thousands of controls launching against an accelerating regulatory environment, six months per control was not a number we could afford.

### Why naive automation does not work here

The first instinct in any LLM-shaped problem is "let a model draft it." That fails the moment you put the output in front of a compliance reviewer.

A monolithic generation gives you a polished but unreviewable document: no record of which clause came from which regulation, no separation between fact and inference, no way to know if Section 3.4 still reflects the source rule after a revision. Compliance work is not about producing prose. It is about producing **defensible prose**: every claim traceable, every change attributable, every revision audit-loggable.

Single-prompt LLM workflows produce the opposite of that. They produce confident-sounding text with no inspectable intermediate state and no story you can tell a regulator about how the document came to be.

Chat-based agents are not better. A growing conversation transcript looks like memory, but it is mostly noise. By turn fifteen, the model is reasoning over its own outputs. A reviewer trying to figure out who changed what and when has to read an entire dialog history to reconstruct it.

The system had to start from a different assumption: **the LLM is a component, not the product.** What ships is the document plus its provenance. The architecture has to reflect that.

## The approach

A four-agent pipeline built with **Strands Agents SDK** on **AWS Bedrock AgentCore**, with explicit state handoffs between agents and a stateful human-in-the-loop refinement layer.

```text
planner  →  drafter  →  critic  →  refiner
                                      ↻
                              human feedback
```

Each agent has a single, narrow job. Each agent reads from explicit state, not from chat history. The pipeline is a series of state transitions, not a conversation.

### Why four agents, not one

The first design instinct in agentic systems is to do everything in one prompt. For high-stakes work, that is usually the wrong shape.

- **Planner** decomposes the incoming control requirement into a structured outline: scope, applicable regulations, controls in force, evidence requirements, and exception clauses. The output is structured JSON, not prose. Every downstream agent reads from the plan, not from a sliding chat history. This makes the plan inspectable. A reviewer can disagree with the planner's decomposition before any drafting begins, which is much cheaper than disagreeing with a finished policy document.

- **Drafter** generates policy text section by section, grounded in retrieved regulatory context. Hybrid retrieval feeds this stage: BM25 over structured control metadata, kNN vector search over unstructured regulatory corpora. Each section gets its own retrieval call so citations stay tight to the text they support. The drafter never sees the whole document at once. It sees the plan and the retrieval results for the section it is currently writing.

- **Critic** runs LLM-as-judge against an explicit rubric: regulatory coverage, citation grounding, internal consistency, and tone calibration. The critic returns **structured findings**, not free-form commentary. A reviewer can read "Section 3.4 missing reference to GDPR Art. 17" as a row in a table. They do not have to parse a paragraph of critique. Findings are addressable, dismissible, and trackable through the refinement cycle.

- **Refiner** integrates human edits. This is where the design gets interesting.

### Human feedback as state, not chat turn

The single most important design decision in the system is that human refinement is **a state transition**, not a turn in a chat history.

Most agentic systems treat human feedback like a message. The user types "make Section 3.2 stricter on data residency" and the model concatenates that onto a growing conversation transcript. This works fine for low-stakes generation. It fails for compliance for two reasons:

1. **Audit trail.** A regulator asks: which specific reviewer made which change to which section, and when? With chat history, the answer is "search through 47 turns and infer." With explicit state, the answer is a query against a versioned document store.

2. **Compounding drift.** Chat-based refinement carries the entire prior context, including prior model mistakes, into every revision. After ten turns, the model is reasoning over its own outputs. Stateful refinement re-runs the relevant agent against the **current canonical state**, not against an accumulated dialog.

In practice, the user's edit is parsed, attributed to a specific section, stored as a new document revision, and passed to the refiner against the updated state. The critic re-validates the refined output before it goes back to the user. The cycle is bounded. The state is queryable. The history is reconstructible.

The schema of human feedback matters as much as the schema of the document. Edits are not free text. They are structured operations against named sections of the document, with an actor, timestamp, intent, and justification. When a regulator, internal auditor, or future reviewer asks why a clause says what it says, the system can show them.

## The infrastructure that made it shippable

The parts that determine whether the system actually ships are not the parts that look impressive in a demo:

- **Idempotency keys** on every agent invocation so retries from transient Bedrock errors do not generate duplicate sections.
- **Versioned state** with full revision history. Any section can be diffed against any prior revision, by any reviewer, at any time.
- **Structured tool calls** instead of free-form output everywhere reviewers consume the result. JSON in, JSON out, until the final document is assembled for human reading.
- **Audit logging** on every state transition: who triggered which agent, what the input state was, what the output state was, which prompt template version was used, which model version ran, and which regulation revisions were in the retrieval index at the time.
- **Retry semantics** with bounded exponential backoff on Bedrock invocations. The distinction between recoverable errors, like rate limits and transient timeouts, and non-recoverable errors, like prompt-level failures and schema validation failures, is explicit at the agent layer.
- **Cost and latency budgets** per agent, instrumented at the invocation level, so we know what each control generation costs and where the time goes.

None of this is the part that makes a demo sing. All of it is the part that determines whether compliance will sign off on using the system at all.

## The results

The common-case authoring cycle dropped from ~6 months to ~2 days. The numbers worth unpacking:

- **Drafting time:** days of human authoring compressed into minutes of agent invocation.
- **Review cycles:** typical four-to-six full handoffs across legal, security, and product reduced to one-to-two focused review cycles.
- **Reviewer time per control:** hours of reading and commenting shifted toward minutes of targeted refinement against an already-validated draft.

The compression does not come from the LLM being smart. It comes from giving reviewers something **inspectable** to react to, with attribution and structure, instead of a wall of prose they have to read top to bottom every cycle. The reviewer experience shifts from reading and critiquing to steering and approving. That shift is most of the win.

The system runs as part of the broader control launch platform powering 2,000+ control launches in 2026, a scale that was structurally impossible at six months per control.

## Tradeoffs and limitations

A few things are worth being honest about.

**The system makes the common case fast, not the hard case.** Novel controls in unfamiliar regulatory territory still need substantial human authoring. The four-agent pipeline handles controls that follow established patterns. The genuinely new ones, ambiguous jurisdictional questions and first-of-their-kind regulations, still get authored by humans, with the agent acting more like a research assistant than a generator.

**LLM-as-judge is not a substitute for human review.** The critic catches a meaningful fraction of issues: missing citations, internal inconsistencies, and regulatory misalignments. It does not catch judgment calls about scope, applicability, or business risk. A human compliance reviewer still signs off on every control before launch. The critic raises the floor on quality. It does not replace the ceiling of human judgment.

**Retrieval recall is the real ceiling.** When the system fails, it usually is not because an agent made a bad reasoning call. It is because the retrieval layer missed a relevant regulation. Improving retrieval through better chunking, legal-text embeddings, and reranking has a higher ceiling than improving the agents themselves. Most model behavior is bottlenecked by what gets retrieved, not what gets reasoned over.

**The system is fragile to upstream changes.** A new regulation, a change in internal control taxonomy, or a Bedrock model version bump can affect output quality in ways that are hard to test against in advance. Continuous evaluation against a held-out set of known-good controls catches the gross failures. Subtle quality regressions are still mostly caught by reviewers downstream.

## What I would do differently

A few things are worth calling out in retrospect.

**Build the critic and rubric first.** The critic was the last agent to land in the implementation, which meant the team spent more time than I would have liked reverse-engineering quality criteria from reviewer feedback after the fact. The cleaner sequence is the opposite: build the critic and its rubric first. Articulating what **good** looks like before you start generating forces a discipline that is hard to recover later. The rubric is also the spec. If you cannot write the rubric, you do not yet understand the problem.

**Invest in the evaluation harness earlier.** Subjective spot-checks scale to about twenty controls. After that, you need a structured eval set with known-good examples, automated regression checks against critic outputs, and the discipline to run the eval on every prompt change, model version bump, and retrieval index update. We built one eventually. We should have built it before we built the agents.

**Treat the schema as the design.** Most of the real design work in the system is not in the prompts. It is in the schemas: the schema of the plan, section, citation, critic finding, and human edit. When the schemas are right, the agents are almost incidental. When the schemas are wrong, no amount of prompt tweaking saves you. Time spent on schema design is the highest-leverage time in the project.

## What this work taught me

Agentic systems are mostly software engineering. The interesting parts, the ones that determine whether the thing actually ships, are the boring ones: state schemas, idempotency, audit logging, retry semantics, and evaluation harnesses. The model itself is a component in a larger system. Treating it that way is the difference between a flashy demo and something a compliance team will actually use.

The other lesson: specialization beats capacity. A small model running a tightly scoped agent is often more reliable than a frontier model running a broad one. The planner does not need to be broadly brilliant. It needs to produce consistent structured output against a stable schema. Match model capacity to the actual difficulty of each agent's job. A pipeline of focused, well-tested agents beats a single sprawling generalist almost every time in production.
