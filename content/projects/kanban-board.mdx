## System intent

This is a compact Kanban application focused on frontend architecture rather than feature count. The goal is to model board state cleanly enough that drag-and-drop behavior, persistence, and responsive rendering do not turn into tangled component state.

It is not trying to compete with Trello. It is a focused UI systems project: state transitions, drag ergonomics, component boundaries, and local persistence.

## Requirements

- Cards should move between columns without corrupting order.
- Board state should survive refresh.
- Components should stay small enough to extend.
- The layout should work at laptop and tablet widths.
- State transitions should be explicit and typed.

## High-level design

The board model owns columns and cards. UI components render from that model and dispatch typed transitions. Persistence is local, which keeps the project simple while still forcing a real serialization boundary.

## Design decisions

**Centralize board updates.** Dragging a card affects at least two columns. Keeping that transition in one place avoids duplicate reorder logic.

**Persist the model, not the DOM.** The stored state is the board object. The UI can change without breaking persistence as long as the model remains stable.

**Keep drag state temporary.** The app should only persist completed moves, not transient hover state.

## What I would improve

The next version would add keyboard drag interactions, undo, and a richer persistence backend. The most important accessibility improvement would be a non-pointer workflow for moving cards between columns.
