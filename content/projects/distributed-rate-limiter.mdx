## System intent

This project implements a distributed sliding-window rate limiter for services that run on multiple app instances. The system needs to make the same decision no matter which instance receives a request, so local process memory is not enough.

The design goal is simple: given a subject key, a limit, and a time window, return an allow-or-throttle decision that is atomic, predictable, and cheap to operate.

## Requirements

- Decisions must be shared across app instances.
- The trim, count, insert, and TTL update must be atomic.
- Old request records should expire without a separate cleanup worker.
- The caller should receive a simple decision and enough metadata to set headers.
- The algorithm should stay understandable for an on-call engineer.

Each rate-limit bucket is represented as a Redis sorted set. Request timestamps are both the score and the member value. On each request, the Lua script removes expired entries, counts the remaining entries, inserts the current timestamp if capacity exists, updates TTL, and returns the decision.

## Design decisions

**Use a sliding window instead of fixed windows.** Fixed windows allow bursts at boundaries. Sliding windows provide smoother behavior and are closer to what users expect from rate limits.

**Use Lua for atomicity.** Multiple app instances can hit the same key concurrently. Doing trim, count, and insert as separate Redis calls creates race conditions. Lua keeps the whole decision in one operation.

**Keep the public API boring.** Most rate-limit complexity should stay behind a small interface. Application code should not need to know the Redis data structure.

## Failure modes

If Redis is unavailable, the service has to choose fail-open or fail-closed per endpoint. A read-heavy endpoint might fail open with degraded headers. A write-heavy or expensive endpoint should probably fail closed. The project keeps that policy outside the core algorithm so callers can make the right product choice.

## What I would improve

The next production step would be a small middleware package with standard `X-RateLimit-*` headers, metrics for blocked requests, and a fallback strategy that can be configured per route.
