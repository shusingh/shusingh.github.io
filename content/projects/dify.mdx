## Change intent

Dify supports realtime collaboration over Socket.IO. In a single-process deployment, Socket.IO's default in-memory manager is enough. Once the application runs across multiple workers or instances, that assumption breaks: events emitted in one process are not automatically visible to clients connected through another process.

This PR adds Redis-backed Socket.IO manager selection when collaboration mode is enabled. When collaboration is disabled, the app keeps the simpler in-memory manager.

## Requirements

- Preserve the existing in-memory Socket.IO manager for non-collaboration deployments.
- Use a Redis-backed manager when collaboration mode is enabled.
- Keep manager selection close to Socket.IO server construction.
- Avoid changing the public Socket.IO event surface.
- Add unit coverage for both manager selection and server construction.

The implementation keeps the Socket.IO setup explicit. The extension checks whether collaboration mode is enabled, constructs the appropriate manager, and passes it into the Socket.IO server. Tests cover the default single-process path and the Redis-backed collaboration path.

## Design decisions

**Do not make Redis mandatory.** Single-process deployments should not pay operational complexity for a manager they do not need.

**Keep topology behind configuration.** Runtime mode should decide whether realtime events need cross-worker coordination. Callers should not need to know the manager implementation.

**Test server construction, not only helper logic.** The meaningful contract is that the Socket.IO server receives the correct manager. Testing only a helper function would miss integration drift.
