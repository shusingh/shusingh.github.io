## System intent

AgentOps RAG is a production-style retrieval platform for agentic applications. The point is not to prove that a model can answer questions from documents. The point is to show the infrastructure an engineering team needs before trusting retrieval in a multi-tenant product: tenant boundaries, async ingestion, replayable failures, grounded answers, evals, traces, rate limits, and benchmarks.

The repo is intentionally explicit. It avoids hiding the runtime behind a large agent framework so the important system boundaries stay visible.

## Requirements

The system is designed around five non-negotiable requirements:

- A tenant must never read another tenant's documents or replay another tenant's failed jobs.
- Ingestion failures must be inspectable and replayable without losing the original request context.
- Answers must be citation-grounded, or the system should refuse instead of inventing support.
- Operators need traces and eval reports, not just server logs.
- Expensive endpoints need rate limits with explicit degraded behavior when Redis is unavailable.

These requirements shape the architecture more than the model choice does.

## High-level design

The write path starts at `POST /documents`. The API validates the JWT, derives tenant ID from claims, stores metadata in Postgres, and enqueues an ingestion job in Redis Streams. The worker chunks the document, indexes chunks into OpenSearch, updates document state, and writes failed jobs to a DLQ with tenant ID, document ID, stage, and retry count.

The read path starts at `POST /ask`. The endpoint applies tenant-scoped rate limits, runs the planner, executes tenant-filtered retrieval, sends retrieved evidence to the critic, and finalizes either a grounded answer or a refusal. Every answer includes a trace ID.

## Design decisions

**Tenant ID comes from JWT claims, not request bodies.** This prevents a client from asking for another tenant by changing payload fields. The same rule applies to document listing, retrieval, and DLQ replay.

**Retrieval is hybrid by default.** BM25 handles exact names, IDs, and policy terms. Vector search handles semantic similarity. The runtime fuses the evidence instead of trusting one retrieval mode for every question.

**Refusal is a product behavior.** If the critic cannot verify citation coverage, the finalizer returns a refusal. That is preferable to a confident answer with weak support.

**DLQ replay is scoped.** Failed ingestion jobs keep tenant and document metadata, so an operator can replay the right work without rerunning the whole corpus.

The eval harness reports answer containment, citation precision, citation recall, refusal accuracy, unsupported claim rate, and latency. Benchmarks report throughput, p50, p95, failure rate, refusal count, and rate-limit blocks. Traces cover HTTP requests, auth validation, ingestion enqueue, worker jobs, chunking, indexing, retrieval, score fusion, model calls, critic decisions, and finalization.

## Tradeoffs

The design favors clarity over framework leverage. A framework could reduce boilerplate, but it would also hide the agent boundary. For a portfolio project, explicit orchestration is a feature because it shows how the runtime behaves under failure.

The next production hardening steps are managed Postgres, Redis, and OpenSearch; Alembic migrations; OTLP trace export; CI eval gates; worker autoscaling; retrieval drift monitoring; and audit logs around tenant onboarding and replay actions.
