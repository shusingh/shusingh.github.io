## Change intent

The embedded terminal in PostHog/code was rendering special characters incorrectly. That usually points to an encoding boundary problem: bytes from a PTY are not the same thing as display-ready strings.

The PR fixes the rendering path by decoding PTY output as UTF-8 before it reaches the terminal renderer, and by aligning the terminal font stack with the app's code font.

## Requirements

- Preserve existing terminal behavior for ASCII output.
- Decode multibyte UTF-8 sequences correctly.
- Keep the renderer from handling raw byte buffers.
- Use the app's code font stack for visual consistency.
- Add tests around the shell service behavior.

The code change is small, but the boundary is important. Terminal rendering should not depend on downstream components guessing how to decode raw process output. Once the service returns valid strings, the renderer can stay focused on display.

## Design decisions

**Decode once at the boundary.** This avoids duplicated decoding logic and keeps malformed output easier to test.

**Do not treat fonts as cosmetic only.** Terminal readability depends on glyph support and spacing. Using the app's code font makes the terminal feel like part of the editor.

**Cover the service with tests.** Encoding bugs regress easily because ASCII happy paths keep passing.
