## Change intent

Agent frameworks rely on schemas to distinguish missing arguments from explicitly provided values. That distinction matters when a tool parameter is both required and nullable.

Before this PR, validated tool arguments could lose explicit `null` values during serialization. The tool call started with a meaningful value, but the invocation boundary treated it like absence. That is a subtle correctness bug: the model or caller did provide the argument, and the tool contract allowed `null`.

This PR preserves explicit `null` values during tool invocation for both direct `FunctionTool.invoke()` calls and automatic function calling.

## Requirements

- Preserve explicit `null` for required nullable parameters.
- Keep missing arguments distinct from present-but-null arguments.
- Apply the same behavior to direct tool invocation and automatic function calling.
- Avoid broad changes to tool schema semantics.
- Add regression coverage for both invocation paths.

The implementation keeps the fix close to the argument dump path. Validated arguments are serialized without dropping explicitly provided nullable fields. Tests cover the direct tool invocation path and the automatic function-calling path so both surfaces preserve the same contract.

## Design decisions

**Treat `null` as a value, not absence.** In tool-calling systems, `null` can be a deliberate argument. Dropping it changes the meaning of the call.

**Patch the shared invocation boundary.** Fixing only direct invocation or only automatic calling would leave two subtly different contracts in the framework.

**Add regression tests at user-visible paths.** The bug is not interesting as an internal serialization detail. It matters because users can define required nullable tool parameters and expect invocation to respect them.
