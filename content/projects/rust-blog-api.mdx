## System intent

This is a deliberately small REST API for blog-style resources. The project exists to practice Rust backend ergonomics: ownership-aware request handling, typed data models, explicit errors, and a clean route/service boundary.

The scope is intentionally constrained. A small API makes it easier to focus on the design pressure Rust applies to backend code.

## Requirements

- Expose predictable REST endpoints for post resources.
- Keep request and response types explicit.
- Return useful errors without panics.
- Make storage replaceable without rewriting route handlers.
- Keep the code small enough that the service shape is easy to inspect.

## High-level design

<FlowDiagram
  nodes={['HTTP request', 'route handler', 'validate payload', 'domain operation', 'HTTP response']}

/>

The API layer handles routing and extraction. Domain types describe posts and validation. Error types map cleanly into HTTP responses. Persistence is intentionally simple in the first version, but the boundary is shaped so a database can be added later.

## Design decisions

**Keep errors explicit.** Rust makes it cheap to model expected failure paths. That is useful for APIs because client-facing failures should be deliberate.

**Avoid framework magic.** Actix provides the web layer, but the core resource model stays plain and easy to test.

**Design for future persistence.** Even when the first version is small, the API should not bake storage details into every handler.

## What I would improve

The next step would be Postgres persistence, migrations, structured logging, and integration tests that lock down the API contract.
