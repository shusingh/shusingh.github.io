## System intent

This project is the smaller Go-based sibling of AgentOps RAG. Its purpose is to practice retrieval infrastructure without the full agent runtime: service boundaries, query handling, OpenSearch integration, metadata persistence, cache paths, and citation assembly.

It is useful as an HLD exercise because it keeps the scope tight. The system does not try to be a complete AI product. It focuses on the retrieval platform underneath one.

## Requirements

- Keep the external API typed and boring.
- Preserve enough metadata to explain why a source was returned.
- Support exact and semantic retrieval paths later.
- Cache repeated query paths without hiding freshness rules.
- Return citations as part of the response contract, not a UI afterthought.

The Go API owns request validation and response shape. OpenSearch owns search and filtering. Postgres stores durable document and query records. Redis is reserved for hot-path cache decisions where repeated queries should not pay full retrieval cost.

## Design decisions

**Citation assembly is a backend concern.** The frontend should not guess which source supports which claim. The API should return citation metadata in a shape that can be rendered directly.

**Metadata is first-class.** Search quality depends on more than embeddings. Document type, tenant, source, version, created time, and indexing state all need a place in the data model.

**Caching needs invalidation rules.** RAG cache entries are dangerous when the corpus changes. This repo treats cache keys as coupled to index version and query normalization.

## Tradeoffs

This project intentionally does less than AgentOps RAG. It is not trying to show auth, DLQ replay, evals, and traces end to end. It is a focused retrieval-platform workbench. The main improvement would be adding an eval set for retrieval quality, because recall failures are the ceiling for any downstream agent.
