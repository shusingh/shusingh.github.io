## System intent

AI Trip Planner is a full-stack application for generating travel itineraries from user preferences. It is intentionally small enough to ship, but it exercises the same boundaries a larger AI product needs: typed UI state, backend-owned model calls, response normalization, latency handling, and deployment.

The important design choice is that the frontend does not call the model provider directly. The Go API owns prompt construction, inference calls, response shaping, and failure handling.

## Requirements

- The UI should collect trip constraints without exposing prompt details.
- The backend should return clean JSON instead of model-shaped text.
- Slow generation should not make the UI feel broken.
- Provider failures should produce useful user-facing states.
- The app should be deployable end to end on simple hosting.

## High-level design

<FlowDiagram
  nodes={['trip form', 'typed request', 'Go prompt builder', 'model call', 'normalize itinerary', 'render plan']}
  caption="The backend translates user constraints into a model request and translates model output back into a UI-friendly itinerary."
/>

The frontend collects destination, dates, interests, budget, and pacing. The Go service builds a structured prompt, calls the inference API, parses the result into an itinerary shape, and returns the response to the UI. The UI only handles product states: loading, success, empty result, and error.

## Design decisions

**Keep inference server-side.** This protects provider details, avoids leaking API keys, and keeps prompt and parsing logic out of the browser.

**Normalize before rendering.** The model can produce messy text. The UI should receive sections, days, activities, and notes in a predictable shape.

**Design for cold starts.** The live demo runs on free hosting, so the first request after idle can be slow. The UI needs a real loading state rather than pretending generation is instant.

## Failure modes

The major failure cases are provider timeout, malformed model output, and free-tier cold start. The app handles these as product states rather than raw exceptions.

## What I would improve

The next version would add saved trips, streaming itinerary generation, editable day cards, and an eval set for response quality. The highest-leverage improvement would be making model output validation stricter before the UI renders it.
