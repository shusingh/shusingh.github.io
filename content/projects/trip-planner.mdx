## Why I built it

I wanted one end-to-end project that exercised the whole stack the way a real product would: a typed frontend, a real backend in a different language, and an inference layer that the backend talks to directly instead of leaking model details into the UI.

A trip planner was a good shape for it. The UX is small enough to actually finish, but the data flow exercises the parts that matter: structured prompts, response shaping, error paths, and rate limiting on a free Render tier.

## The stack

- **Frontend.** Vite + React + TypeScript. Single-page app, typed API client, no UI framework noise.
- **Backend.** Go. The API does the LLM call, normalizes the response into a typed itinerary shape, and returns clean JSON.
- **Inference.** Hugging Face Inference API for itinerary generation. Wrapped behind the Go service so the frontend never sees model-specific details.

## What it taught me

The interesting parts were on the seams. How to keep the model's freeform output from polluting the frontend's type system. Where to push retry and rate limiting (backend, not browser). How to keep the UI honest about latency when generation takes 5+ seconds on a cold model.

## Status

Released and deployed on Render. The live demo runs on a free tier, so the first request after idle is slow while the service spins up.
