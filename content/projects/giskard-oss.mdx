## Change intent

Giskard helps teams test ML and LLM systems through reusable checks. Structured JSON output is now a common contract for agents, tool calls, and model pipelines, but the check suite did not have a first-class JSON validity check.

This PR adds `JsonValid`, a check that validates both raw JSON strings and already-parsed JSON-compatible values. It also supports optional JSON Schema validation.

## Requirements

- Accept raw JSON strings and JSON-compatible Python objects.
- Distinguish invalid JSON from schema validation failure.
- Treat malformed schemas as configuration errors, not data failures.
- Add test coverage for syntax-only and schema-backed validation.
- Keep the API consistent with the existing Giskard check patterns.

The implementation validates syntax first. If a schema is provided, it validates the parsed value against that schema. Invalid user data produces a validation failure. Invalid schema configuration produces a configuration error so teams do not mistake a broken check for a broken model output.

## Design decisions

**Support parsed objects.** Many pipelines parse JSON before checks run. Requiring teams to serialize back to strings would make the check awkward to compose.

**Use JSON Schema for shape validation.** Syntax validity only answers "is this JSON?" Schema validation answers "does this match the contract?"

**Keep error classes meaningful.** Debugging LLM output requires knowing whether the model failed, the schema failed, or the pipeline wiring failed.
